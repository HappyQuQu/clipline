# Clipline MVP Scope

## 1. 文档目的

本文档定义 Clipline 第一版 MVP 的开发范围、优先级、验收边界和暂不实现能力。

MVP 的目标不是做完整 NVR，也不是做复杂视频平台，而是验证一个核心闭环：

```text
用户在 Web UI 中新建录像源 -> 系统扫描已有视频片段 -> 用户按日期时间线回看 -> 用户跨片段导出视频
```

## 2. MVP 核心目标

- 用户可以用 Docker Compose 启动 Clipline。
- 用户可以在 Web UI 中新建一个或多个录像源。
- 用户只需要填写源名称，并选择一个容器内可访问的视频目录。
- 系统可以扫描该目录下的视频文件并建立索引。
- 用户可以按源、日期和时间查看录像片段。
- 用户可以播放单个片段，并在片段结束后自动切换到下一片段。
- 用户可以选择一段真实时间范围，并导出一个完整 MP4 文件。

## 3. P0 必须完成

P0 是 MVP 可用闭环，缺少任一项都不应发布。

### 3.1 部署与启动

- 提供 Dockerfile。
- 提供 Docker Compose 示例。
- 容器内包含后端、前端静态文件、FFmpeg、ffprobe。
- 应用数据写入 `/app/data`。
- 录像目录只读挂载到 `/recordings`。
- 重启容器后 SQLite 和导出记录不丢失。

### 3.2 源管理

- 设置页支持新建源。
- 新建源时填写源名称。
- 新建源时选择 `/recordings` 下的视频目录。
- 系统自动生成源 ID。
- 系统默认使用应用时区。
- 系统默认使用自动时间识别规则。
- 支持启用、停用源。
- 支持手动触发重新扫描。
- 禁止保存不存在、不可读、超出录像根目录的路径。
- 禁止多个源使用完全相同的路径。

### 3.3 扫描与索引

- 支持扫描 `.mp4`、`.mov`、`.mkv`、`.avi`、`.ts`、`.m4v`。
- 使用 ffprobe 获取视频时长、编码、分辨率、帧率等元数据。
- 根据文件名、目录名、视频 metadata、mtime 推断片段开始时间。
- 将扫描结果写入 SQLite。
- 同一文件大小和 mtime 未变化时，不重复 ffprobe。
- 损坏文件不影响整个扫描任务。
- 正在写入或不稳定的文件标记为 pending，后续重扫。

### 3.4 回看页

- 顶部支持源选择和日期选择。
- 主区域展示视频播放器。
- 展示当天时间线。
- 时间线显示有录像区间和断档。
- 展示当天片段列表。
- 点击片段可以播放。
- 点击时间线有录像区域可以定位到对应片段。
- 当前播放时间能映射到真实时间线。

### 3.5 播放

- 使用浏览器原生 HTML video 播放。
- 支持 Range 请求。
- 支持播放、暂停、音量、静音、全屏。
- 当前片段播放结束后自动播放下一个片段。
- 如果两个片段之间存在断档，界面显示断档提示后跳到下一片段。
- 浏览器不支持的编码显示明确提示，仍允许导出。

### 3.6 导出

- 用户可以选择开始时间和结束时间。
- 后端查询与时间范围相交的片段。
- 支持单文件内导出。
- 支持跨两个或多个文件导出。
- 默认提供快速导出。
- 提供精准导出选项。
- 导出任务有 queued、running、completed、failed 状态。
- 前端可以查看导出进度。
- 导出完成后可以下载。
- 导出失败时显示明确错误。
- 导出区间包含断档时，默认跳过断档，只导出实际存在的录像内容。
- 导出区间包含断档时，前端必须提示输出视频时长可能短于选择区间。
- 临时文件在任务结束后清理。

## 4. P1 应该完成

P1 可以显著改善体验，但不阻断 MVP 发布。

- 每个视频生成一张中间帧缩略图。
- 片段列表懒加载缩略图。
- 时间线支持基础缩放。
- 支持按是否可播放过滤。
- 支持按是否存在断档过滤。
- 设置页展示 FFmpeg 状态、SQLite 状态、源目录可读状态。
- 设置页展示缩略图缓存大小和导出缓存大小。
- 支持清理导出缓存。

## 5. P2 可以延后

P2 不进入第一版开发范围。

- AI 人车检测。
- 运动检测。
- 多摄像头同步播放。
- 实时摄像头流。
- HLS 预览转码。
- 移动端 PWA。
- 多账号权限。
- WebDAV、SMB、S3 内置挂载。
- 视频删除和生命周期管理。
- 地图轨迹。
- OCR 时间戳识别。
- 插件式时间解析规则。

## 6. MVP 非目标

- 不接管摄像头。
- 不负责实时直播。
- 不负责录像计划。
- 不管理摄像头码流。
- 不做完整 NVR 权限、告警和设备管理。
- 不修改、移动、删除源视频文件。
- 不强依赖 AI 检测。
- 不提供复杂剪辑能力。

## 7. 第一版产品决策

- 源管理在 Web UI 中完成。
- 用户只需要理解“源名称”和“视频目录”。
- Docker 只负责把宿主机录像目录只读映射到容器内 `/recordings`。
- 系统内部自动处理源 ID、时区、时间解析规则。
- 默认局域网裸访问，不启用登录。
- 默认快速导出，同时提供精准导出。
- SQLite 作为唯一数据库。
- 后台任务使用应用内 worker，不引入 Redis。

## 8. 发布验收清单

### 8.1 部署验收

- 使用 Docker Compose 可以启动应用。
- 访问 Web UI 不报错。
- `/app/data` 挂载后数据可以持久化。
- `/recordings` 只读挂载后可以被应用读取。

### 8.2 源管理验收

- 用户可以新建“门口摄像头”源。
- 用户可以选择 `/recordings/front-door` 作为视频目录。
- 保存成功后自动触发首次扫描。
- 选择不存在路径时保存失败。
- 选择 `/app/data` 等非录像根目录路径时保存失败。
- 重复选择同一路径时保存失败。

### 8.3 扫描验收

- 给定 100 个 MP4 文件，系统可以扫描并建立索引。
- 同一目录再次扫描时，不重复处理未变化文件。
- 损坏文件被记录为 failed，不影响其他文件。
- 最近仍在写入的文件不会被错误当作完整视频。

### 8.4 回看验收

- 同一天多个片段能按真实时间排序。
- 时间线能显示有录像区间。
- 时间线能显示断档。
- 点击任意有录像时间点，可以定位到正确片段。
- 点击片段后播放器开始播放。

### 8.5 播放验收

- 浏览器可直接播放 H.264 MP4。
- 当前片段结束后自动播放下一片段。
- 下一片段不连续时显示断档提示。
- 不支持在线播放的编码显示可理解错误，不影响页面整体使用。

### 8.6 导出验收

- 单文件内导出生成完整 MP4。
- 跨两个文件导出生成完整 MP4。
- 跨多个文件导出按时间顺序合并。
- 包含断档的区间导出时，输出视频跳过断档并给出明确提示。
- 快速导出失败时提示用户改用精准导出。
- 导出完成后可以下载。

## 9. 主要风险

### 9.1 时间推断错误

影响：

- 时间线排序错误。
- 点击时间定位错误。
- 跨片段导出错误。

对策：

- MVP 内置常见文件名时间格式识别。
- 保存推断时间来源。
- 无法可靠识别时使用 mtime 减视频时长兜底。
- 后续提供自定义时间规则。

### 9.2 H.265 无法在线播放

影响：

- 摄像头常见视频无法在浏览器直接播放。

对策：

- 扫描时标记 `needs_transcode`。
- 页面明确提示暂不支持在线播放。
- 仍允许精准导出转码。
- 后续支持 HLS 转码预览。

### 9.3 快速导出不精确

影响：

- `-c copy` 受关键帧影响，裁剪点可能偏移。

对策：

- UI 明确区分快速导出和精准导出。
- 证据用途推荐精准导出。
- 快速导出失败时提示改用精准导出。

### 9.4 大目录首次扫描慢

影响：

- NAS 首次扫描可能耗时较长。

对策：

- 扫描任务后台执行。
- UI 显示扫描进度。
- 增量扫描避免重复 ffprobe。
